# Ознакомление с базовымы командами в git and github
План:
1. > git init
2. > git add .
3. > git commit -m "description of commmit"
4. > git branch -M main
5. > git clone
6. > git remote and origin
7. > git push
## 1. Git init 
git init — это команда в системе контроля версий Git, которая используется для инициализации нового репозитория. Когда вы выполняете эту команду в директории, Git создаёт подкаталог с именем .git, который содержит все необходимые файлы и структуры данных для отслеживания изменений в вашем проекте. Вот основные шаги, как использовать git init:
1. ___Создание новой директории (если у вас её ещё нет):___\
	```
	mkdir my_project
	```
	```
	cd my_project
	```
2. ___Инициализация репозитория:___\
	```
	git init
	```
## 2. Git add
git add — это команда в системе контроля версий Git, которая используется для добавления изменений в индекс (или staging area). Индекс — это промежуточное хранилище, где вы можете подготовить изменения перед их коммитом. Вот основные команды.
1. ___Добавление отдельных файлов:___
	```
	git add имя_файла
	```
2. __Добавление всех изменений:___
   ```
   git add .
   ``` 
   или 
   ```
   git add -A
   ```
3. ___Добавление изменений в конкретной директории:___
   ```
   git add путь/к/директории/
   ```
4. ___Проверка статуса:___
	```
	git status
	```
	Эта команда покажет, какие файлы были изменены, добавлены или удалены, а также какие изменения готовы к коммиту.
## 3. Git commit
git commit — это команда в системе контроля версий Git, которая используется для сохранения изменений в локальном репозитории. Когда вы выполняете git commit, все изменения, которые были добавлены в индекс с помощью команды git add, фиксируются в истории проекта. Каждый коммит имеет уникальный идентификатор (SHA-1), который позволяет отслеживать изменения.
1. ___Фиксация изменений:___\
   ```
   git commit -m "Ваше сообщение коммита"
   ```
	Здесь -m указывает, что за ним следует сообщение коммита, которое должно описывать сделанные изменения. Хорошая практика — писать информативные сообщения, чтобы другие разработчики (или вы сами в будущем) могли понять, что было изменено.
2. ___Фиксация всех изменений:___\
    Если вы хотите зафиксировать все изменения, которые были добавлены в индекс, просто выполните команду git commit с сообщением. Если вы не добавили файлы в индекс с помощью git add, то коммит не будет выполнен.
3. ___Коммит без сообщения:___\
   Если вы не укажете сообщение с помощью -m, Git откроет текстовый редактор (по умолчанию это может быть Vim или другой редактор), где вы сможете ввести сообщение коммита.
4. ___Просмотр истории коммитов:___\
   После выполнения коммита вы можете просмотреть историю коммитов с помощью команды:\
	```
	git log
	```
	Эта команда покажет список всех коммитов в текущем репозитории, включая их идентификаторы, авторов и сообщения.
5. ___Отмена последнего коммита:___\
   Если вы хотите отменить последний коммит, но сохранить изменения в рабочем каталоге, используйте:\
	```
	git reset --soft HEAD~1
	```
	Если вы хотите отменить коммит и удалить изменения, используйте:\
	```
	git reset --hard HEAD~1
	```

Команда git commit является одной из ключевых в Git, так как она позволяет сохранять и отслеживать изменения в проекте. Правильное использование коммитов помогает поддерживать чистую и понятную историю изменений.
## 4. Git branch
git branch — это команда в системе контроля версий Git, которая используется для управления ветками в репозитории. Ветки в Git позволяют вам работать над различными версиями проекта параллельно, не мешая основной (обычно это ветка main или master).

Вот основные аспекты работы с командой git branch:
1. ___Просмотр всех веток:___\
   Чтобы увидеть список всех веток в вашем репозитории, выполните команду:
   ```
   git branch
   ```
   Текущая ветка будет отмечена звездочкой (*).

2. ___Создание новой ветки:___\
   Чтобы создать новую ветку, используйте команду:
	```
	git branch имя_ветки
	```
	Например, чтобы создать ветку с именем feature-xyz, выполните:
	```
	git branch feature-xyz
	```
3. ___Переключение между ветками:___\
	Чтобы переключиться на другую ветку, используйте команду:
	```
	git checkout имя_ветки
	```
	Например:
	```
	git checkout feature-xyz
	```
	Также можно создать новую ветку и сразу переключиться на неё с помощью одной команды:
	```
	git checkout -b имя_ветки
	```
4. ___Удаление ветки:___\
	Чтобы удалить ветку, используйте команду:
	```
	git branch -d имя_ветки
	```
	Если ветка не была слита и вы хотите принудительно удалить её, используйте флаг -D:
	```
	git branch -D имя_ветки
	```
5. ___Слияние веток:___\
	Чтобы слить изменения из одной ветки в другую, сначала переключитесь на целевую ветку (например, main), а затем выполните команду:
	```
	git merge имя_ветки
	```
	Это объединит изменения из указанной ветки в текущую.
6. ___Создание веток для работы над фичами:___\
	Использование веток для разработки новых функций (фич) является хорошей практикой. Это позволяет вам изолировать изменения до тех пор, пока они не будут готовы к слиянию с основной веткой.

Команда git branch и работа с ветками в целом являются важными аспектами управления проектами в Git, позволяя разработчикам эффективно работать над различными задачами одновременно и поддерживать чистую историю изменений.

## 5. Git clone
git clone — это команда в системе контроля версий Git, которая используется для создания локальной копии удалённого репозитория. Эта команда позволяет вам загрузить все файлы, историю коммитов и ветки из удалённого репозитория на ваш компьютер, что делает её основным способом работы с существующими проектами.\
Вот основные аспекты работы с командой __git clone__:

1. ___Клонирование репозитория:___\
   Чтобы клонировать репозиторий, вам нужно знать его URL. Команда имеет следующий синтаксис:
   ```
   git clone <URL_репозитория>
   ```
   Например, чтобы клонировать репозиторий с GitHub, вы можете выполнить:
   ```
   git clone https://github.com/имя_пользователя/имя_репозитория.git
   ```
2. ___Создание директории:___\
   По умолчанию git clone создаёт новую директорию с именем репозитория в текущем каталоге. Если вы хотите указать другое имя для директории, вы можете сделать это следующим образом:
   ```
   git clone <URL_репозитория> <имя_директории>
   ```
   Например:
   ```
   git clone https://github.com/имя_пользователя/имя_репозитория.git my_project
   ```
3. ___Клонирование с определённой веткой:___\
   Если вы хотите клонировать репозиторий и сразу переключиться на определённую ветку, вы можете использовать флаг -b:
   ```
   git clone -b имя_ветки <URL_репозитория>
   ```
4. ___Клонирование с использованием SSH:___\
   Если у вас настроен SSH-доступ к удалённому репозиторию, вы можете использовать SSH-URL для клонирования:
   ```
   git clone git@github.com:имя_пользователя/имя_репозитория.git
   ```
5. ___Обновление клонированного репозитория:___\
   После клонирования вы можете использовать команду __git pull__, чтобы обновить вашу локальную копию с изменениями из удалённого репозитория.

Команда git clone является основным инструментом для начала работы с проектами, находящимися под управлением Git, и позволяет разработчикам быстро и легко получать доступ к коду и истории изменений.
## 6. Git remote and origin
### Git remote
git remote — это команда в Git, которая используется для управления удалёнными репозиториями, связанными с вашим локальным репозиторием. Удалённые репозитории — это версии вашего проекта, которые находятся на других серверах, обычно в интернете, например, на GitHub, GitLab или Bitbucket.
1. ___Просмотр удалённых репозиториев:___\
   Чтобы увидеть список всех удалённых репозиториев, связанных с вашим локальным репозиторием, используйте команду:
   ```
   git remote -v
   ```
   Эта команда покажет имена удалённых репозиториев и их URL, как для получения (fetch), так и для отправки (push).

2. ___Добавление удалённого репозитория:___\
   Чтобы добавить новый удалённый репозиторий, используйте команду:
   ```
   git remote add <имя> <URL>
   ```
   Например:
   ```
   git remote add origin https://github.com/имя_пользователя/имя_репозитория.git
   ```
   Здесь origin — это стандартное имя для основного удалённого репозитория, но вы можете использовать любое другое имя.

3. ___Удаление удалённого репозитория:___\
   Чтобы удалить удалённый репозиторий, используйте команду:
   ```
   git remote remove <имя>
   ```
   Например:
   ```
   git remote remove origin
   ```
4. ___Переименование удалённого репозитория:___\
   Если вам нужно изменить имя удалённого репозитория, используйте команду:
   ```
   git remote rename <старое_имя> <новое_имя>
   ```
   
### Git origin
origin — это стандартное имя, которое Git автоматически присваивает удалённому репозиторию при его клонировании. Это имя используется для обозначения основного удалённого репозитория, с которым вы работаете. Например, когда вы выполняете команду git push origin main, вы отправляете изменения из вашей локальной ветки main в удалённый репозиторий, связанный с именем origin.
1. ___Отправка изменений в удалённый репозиторий:___\
   ```
   git push origin main
   ```
2. ___Получение изменений из удалённого репозитория:___\
   ```
   git pull origin main
   ```
3. ___Клонирование репозитория:___\
   При клонировании репозитория origin создаётся автоматически и указывает на удалённый репозиторий, откуда вы клонировали проект.

Работа с git remote и понимание концепции origin являются важными аспектами управления проектами в Git, так как они позволяют вам взаимодействовать с удалёнными репозиториями и синхронизировать изменения.

## 7. Git push
git push — это команда в системе контроля версий Git, которая используется для отправки ваших локальных коммитов в удалённый репозиторий. Эта команда позволяет вам синхронизировать изменения, сделанные в вашем локальном репозитории, с удалённой версией проекта, чтобы другие пользователи могли видеть и использовать эти изменения.
1. ___Отправка изменений:___\
   Чтобы отправить изменения из вашей текущей ветки в удалённый репозиторий, используйте команду:
   ```
   git push <имя_удалённого_репозитория> <имя_ветки>
   ```
   Например, чтобы отправить изменения в ветку main удалённого репозитория с именем origin, выполните:
   ```
   git push origin main
   ```
2. ___Отправка всех веток:___\
   Если вы хотите отправить все локальные ветки в удалённый репозиторий, используйте флаг --all:
   ```
   git push origin --all
   ```
3. ___Отправка тегов:___\
   Чтобы отправить теги в удалённый репозиторий, выполните команду:
   ```
   git push origin --tags
   ```
4. ___Настройка upstream:___\
   Если вы хотите установить отслеживание для вашей локальной ветки, чтобы в дальнейшем можно было использовать более короткую команду git push, выполните:
   ```
   git push -u origin имя_ветки
   ```
5. ___Обработка конфликтов:___\
   Если в удалённом репозитории есть изменения, которые отсутствуют в вашем локальном репозитории, Git может заблокировать git push и предложить сначала выполнить git pull, чтобы синхронизировать изменения. В этом случае вам нужно будет разрешить конфликты, если они возникнут, и снова выполнить git push.
6. ___Принудительная отправка:___\
   Если вы хотите принудительно отправить изменения, даже если это приведёт к перезаписи истории в удалённом репозитории, используйте флаг --force:
   ```
   git push --force origin имя_ветки
   ```
   Будьте осторожны с этой командой, так как она может привести к потере данных в удалённом репозитории.

Команда git push является важным инструментом для совместной работы над проектами, так как она позволяет разработчикам делиться своими изменениями и синхронизировать работу с другими участниками команды.
